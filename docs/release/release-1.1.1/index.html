リリース1.1.1

リリース1.1.1の変更点

  • 主な新機能
  • その他の変更
  • バグ修正
  • 既知の問題

主な新機能

新機能はありません。

その他の変更

  • イングレス・ルートの編集メカニズムが改訂されました:
    • ルート編集ダイアログを開く前に、ドメインのKubernetesネームスペースから既存のサービス・データをフェッチ中にビジー・ダイアログが追加されました。
    • 値が選択されていない場合にこれらのフィールドの現在の状態を明確にするために、Target ServiceおよびTarget Portフィールドにプレースホルダーが追加されました。
    • Target Serviceフィールドが変更され、選択したポートが選択した新しいサービスに対して有効でない場合、Target Portフィールドの選択をリセットするロジックが追加されました。
  • 使いやすさを向上させるため、ドメイン・ページのイメージ・ページおよび関連フィールドを再加工しました。 これらの変更により、新しいイメージを作成する必要がないユーザーは、イメージ・ページを完全にバイパスできます。
    • 変換されたUse Auxiliary ImageおよびCreate New Auxiliary Imageは、3つのオプションをより明確にするためにラジオ・ボタンに切り替えます。
    • Image TagおよびImage Registry Addressフィールドをイメージ・ページのタブから新しいイメージを作成しないときにドメイン・ページに移動しました。
    • Create New Primary ImageおよびAuxiliary Image Configurationコントロールをイメージ・ページからドメイン・ページにレプリケートしました。
    • 新しいイメージの作成時にイメージ・ページに戻るようユーザーに指示するナビゲーション・ボタンをドメイン・ページに追加しました。
  • バンドルされたWebLogic Deploy Tooling 2.1.0。 詳細は、「WDTリリース・ノート」を参照してください。
  • Domain in ImageおよびDomain in PVのユース・ケースでより適切に動作するように、加工されたモデル・アクション。
    • WebLogic Deploy Tooling 2.1.0に追加された新しいターゲット・タイプを使用する拡張されたPrepare Modelアクション。これにより、モデルの準備がModel in Image、Domain in ImageおよびDomain in PVのユースケースで適切に機能するようになります。
    • 以前にValidate ModelアクションとPrepare Modelアクションの両方を使用できないDomain in PVチェックを削除しました。
  • Domain in PVを使用して、プロジェクトにモデルがある場合またはモデルがない場合のユースケースをサポートする場合に編集可能な「拡張ドメイン」ページCluster表。
  • すべてのaria-labelフィールドをi18nリソース・バンドルを使用するように変換しました。
  • webuiユニット・テストをインストーラから削除しました。

バグの修正

  • プロジェクト・ファイルを別のディレクトリに保存するときに「名前を付けて保存」で問題を解決しました。
  • 保留中の変更が存在しない場合、プロジェクトがアーカイブ・ファイルのエントリを失っていた「別名保存」の問題を解決しました。
  • プロジェクト・ファイルからウィンドウへのマッピングにより、資格証明マネージャが不要に再割当てされた問題の解決。
  • Linux VMでデスクトップ・マシンとして使用されていない場合にインストールの問題を解決するために、RPMインストーラの依存関係を追加しました。
  • ネイティブのmacOS ICNSアイコンを追加して、macOSの符号化されたアイコンを解決しました。
  • ルートが存在しない場合の上書き警告を排除するために、既存のイングレス・ルート検出の問題を解決しました。

既知の問題

  • WindowsでWKT UIアプリケーションを実行する場合、イメージ・ビルダー・ツール(DockerまたはPodman)もWindowsで直接実行可能である必要があります。 たとえば、現在、WindowsでWKT UIアプリケーションを実行し、Windows Subsystem for Linux (WSL2)でPodmanを実行するためのサポートはありません。 ただし、WSL2バックエンドでDocker Desktop for Windowsを実行すると、dockerコマンドがWindowsで直接実行可能であるため(WSL2をコールする必要はありません)。 WindowsでPodmanを使用する必要がある場合は、Windowsリモート・クライアントのダウンロード、インストールおよび構成の詳細は、https://podman.io/blogs/2021/09/06/podman-on-macs.htmlおよびhttps://podman.io/blogs/2020/09/02/running_windows_or_mac.htmlのPodmanブログ・エントリを参照してください。

  • Linuxでは、アプリケーションはインストールされているlibGLに依存します。libGLは現在、rpm(またはdeb)インストーラの依存関係リストにリストされていません。 したがって、パッケージ・マネージャを使用してlibGLをインストールする必要があります。 例えば: 

      sudo yum install libGL

  • Linuxマシンでアプリケーションを実行し、Windowsマシンで表示しようとする場合は、Xming Xサーバーを使用しないでください。 Electron 13.x以降を使用するアプリケーションが動作しないバグ(OpenGLサポートが前提)が発生する可能性があります (Microsoft VS Codeも動作しないなど)。

  • アプリケーションは、サイズが2 GB未満のアーカイブ・ファイルの処理に制限されます。
